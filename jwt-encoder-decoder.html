Reference

① Header

Contains metadata about the token: the signing algorithm being used (e.g. HS256) and the token type, which is almost always "JWT".

② Payload

Holds the claims — a JSON object encoded with Base64Url. Common claims include iss, sub, aud, iat, and exp.

③ Signature

A cryptographic string generated from the header and payload, used to verify the token hasn't been tampered with.

🔍 Common use cases

Debugging authentication issues

Decoding reveals claims like exp, iss, aud, and roles — helping you spot expired tokens, wrong issuers, or missing permissions before they break a workflow.

Inspecting token claims

Developers decode access tokens at runtime to populate UI features or enforce backend access control based on roles and scopes.

⚠️ Security considerations

Signature verification

Never trust decoded data alone — always verify the signature to confirm a token wasn't forged or altered.

Risks of decoding unverified tokens
  • Injection of false claims
  • Expired or replayed tokens
  • Forged user roles or escalated privileges
