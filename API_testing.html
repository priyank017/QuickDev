Params are appended to the URL as a query string when they're checked on.

Content-Type: json Accept: json User-Agent

Whatever you pick here is folded into the request's headers (or query string, for API keys set to send that way) right before sending — it isn't stored separately.

About mutual TLS (2-way SSL) here. A client certificate is presented during the TLS handshake, before any JavaScript runs — browsers deliberately don't let a web page attach one to an individual request. There's no way around that from inside any browser-based tool, this one included.

Three ways to actually test an mTLS endpoint:
1. Turn on "Route through local relay" above the address bar. The relay is a real local process, not a browser page, so it attaches the cert/key you enter below to a genuine mTLS handshake — this is the closest thing to "it just works" here.
2. Install the certificate in your OS or browser — it'll handle the handshake automatically for direct (non-relay) requests too.
3. Use the generated cURL command (see the "cURL" tab in the response panel) — no browser involved at all.

These downloads happen entirely in your browser — pasted text turns straight into a file and is never sent anywhere. Save them next to where you'll run the cURL command.

Install this certificate in your OS or browser instead →
  1. macOS: combine cert + key into a .p12 (e.g. via openssl pkcs12 -export), then double-click it to add to Keychain Access.
  2. Windows: convert to .p12, then Settings → Manage user certificates → Personal → Import.
  3. Firefox: Settings → Privacy & Security → View Certificates → Your Certificates → Import (Firefox keeps its own store, separate from the OS).
  4. Chrome / Edge / Safari: use the OS store above — they read from it directly.
